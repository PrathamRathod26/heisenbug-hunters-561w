# database/

PostgreSQL bootstrap artifacts for the single-VM Docker deployment. Everything
here is mounted read-only into the stock `postgres:17-alpine` image — no custom
Dockerfile, no image rebuilds when tuning changes.

## Layout

```
database/
├── init/                    # bind-mounted at /docker-entrypoint-initdb.d:ro
│   ├── 001-extensions.sql   # enable pgcrypto, pg_stat_statements
│   └── 002-roles.sql        # least-privilege claims_app login role
└── conf.d/                  # bind-mounted at /etc/postgresql/conf.d:ro
    └── postgresql.conf      # custom tuning, appended via include_dir
```

## Init scripts (`init/`)

Files are executed in lexicographic order **only on the first boot of an empty
data volume** — that is, when `/var/lib/postgresql/data` is empty. Once the
cluster is initialized, Postgres skips this directory entirely, even on
container restart.

Consequences:

- **Do not put evolving schema here.** Domain DDL and seed data live in the
  backend's Liquibase changelogs (`backend/src/main/resources/db/changelog/`)
  so they run on every boot and are versioned with application code.
- **Adding a new script does not apply it to an existing volume.** To re-run
  init on an existing deployment, either run the SQL manually, wrap it in a
  Liquibase changeset, or destroy and recreate the `postgres-data` volume
  (destructive — data loss).
- **Scripts must be idempotent-safe on fresh volumes.** Use `IF NOT EXISTS` /
  `CREATE OR REPLACE` defensively in case they ever get re-run by hand.

## Config overrides (`conf.d/`)

`docker-compose.yml` starts Postgres with
`-c include_dir=/etc/postgresql/conf.d`, so any `*.conf` file in this
directory is appended after the stock `postgresql.conf`. Last value wins,
which is how we override defaults without editing the base config.

Changes take effect on container restart (`docker compose restart postgres`).
No rebuild needed.

## What does NOT go here

- Liquibase changelogs — `backend/src/main/resources/db/changelog/`
- Entity DDL — generated by Hibernate / captured by Liquibase
- `pg_dump` snapshots — out-of-band backups, not committed
- Application credentials — `.env` (see `.env.example`)
